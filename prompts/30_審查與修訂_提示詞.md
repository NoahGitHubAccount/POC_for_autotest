# 30 — 審查與修訂

## 使用時機
spec md 已 review 但發現問題（AC 描述不清、有遺漏條件、與實測不符），或 test 檔已產出但要依新發現修訂。

## 必填參數
- **修訂目標**：spec md 路徑 / test 檔路徑（可多份）
- **修訂依據**：使用者 review 意見（自然語言或條列）/ 新解碼的 src 證據 / 新發現的踩坑
- **冪等規則**：spec md 的 `<!-- AI-MANAGED -->` 區塊可改、`<!-- USER-EDIT -->` 區塊不動

## 期望輸出
- 修訂後的檔案（內容 + 變更紀錄）
- 同時更新對應檔案的 `最後更新` frontmatter
- 一段「修訂摘要」（給人類 review 用）

## 標準流程

### 模式 A：spec 修訂

**前提**：你有具體的修訂依據（不是腦補），且改動只在 `<!-- AI-MANAGED -->` 區塊內。

1. **讀現況**：完整讀目標 spec md
2. **比對差異**：列出「現有 AC vs 新依據」的具體 diff（在心裡或 todo）
3. **動刀**：
   - AC 區塊：直接改寫
   - 待釐清：把已解決的條目移除或註明「→ 已釐清，見 §X」
   - **絕不**碰 USER-EDIT 區塊
4. **加變更紀錄**：
   ```markdown
   <!-- AI-CHANGELOG START -->
   - 2026-05-08 修訂 AC<n>：<摘要>（依據：<src 路徑 / 使用者意見 / 實測>）
   <!-- AI-CHANGELOG END -->
   ```
5. **更新 frontmatter**：`最後更新: 2026-05-08`、`ai_generated_at: 2026-05-08T...`

### 模式 B：test 修訂

**前提**：spec 改了 → test 要跟；或 test 自身發現問題（selector 失效、xfail reason 不對、斷言過鬆）。

1. **影響面分析**：
   - spec 改 AC<n> → 找對應 `test_AC<n>_*` 函式
   - selector 改 → 找所有用該 Page Object 的 test
2. **修法優先級**：
   - 改斷言（`<=` 改 `<`、加新 assertion）→ 第一選擇
   - 改 selector → 確認源於 src，不對 live DOM 試
   - 加 xfail / skip → 必須附完整 reason，不能只寫 "pending"
3. **不要做的事**：
   - ❌ 把 test 改成 `assert True` 強綠
   - ❌ 把斷言條件放寬到「永遠成立」（例 `>=0`）
   - ❌ 用 `try/except` 吞失敗
   - ❌ 直接刪掉 fail 的 test 案例
4. **變更紀錄**：在檔案頂部 docstring 加一行「<日期>：<改了什麼>」

### 模式 C：lib/selectors.py 修訂

**前提**：DOM 變了 / 新解碼出更穩的 selector / 舊 selector 在某情境失效。

1. **保留既有方法**：除非確認沒人用，否則不刪舊方法
2. **新增為主**：加新 method，舊的標 deprecated（comment 即可）
3. **解碼證據**：在 method docstring 引用 src 路徑與行號
   ```python
   @staticmethod
   def organizer_input(page: Page) -> Locator:
       """主辦單位 MultiSelect。
       依據：src/.../STFilter.vue L42-L60，label='主辦單位' 包 .p-multiselect。
       """
   ```

## 審查 checklist（修訂前自我檢）

- [ ] 修訂依據明確（不是腦補）
- [ ] AI-MANAGED 區塊與 USER-EDIT 區塊邊界清楚
- [ ] 變更紀錄寫了
- [ ] frontmatter `最後更新` 改了
- [ ] 對應的下游檔（test 對 spec / lib 對 test）也檢過

## 與其他提示詞的關係

| 提示詞 | 何時轉到此 | 何時從此轉出 |
|---|---|---|
| `10_PDF轉規格` | 第一次產 spec → 不來這 | 產完進到 review 才用本檔 |
| `20_生成測試案例` | 寫完 test 發現 spec 描述不準 → 來這修 spec | 修完 spec → 回 20 重生對應 test |
| `40_執行測試與排錯` | 跑 pytest 紅了 + 確認是 spec/AC 描述問題 | 修完回 40 重跑 |

## 反模式

- ❌ **「全部重生」**：不要整份 spec / test 重寫覆蓋。冪等規則的 AI-MANAGED 區塊邊界存在就是為了避免這個
- ❌ **靜默修訂**：改了沒寫變更紀錄
- ❌ **跳過 USER-EDIT**：使用者填的內容是事實，AI 不知情
- ❌ **改 USER-EDIT**：永遠不要
