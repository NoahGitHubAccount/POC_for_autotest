## 2026-05-14｜filter() 後的 chained locator count() 在 data:svg img 上不穩定

- **現象**：`page.locator("aside").filter(has_text="...").first.locator("ul li a").count()` 回傳 0，但同個 aside 裡的 `get_by_text("...")` 可以找到文字。
- **原因**：`filter()` 產生的 Locator 在後續 chained CSS selector 的 scope 計算上有時不符預期（特別是包含 `data:svg` img 的結構），但 `get_by_text` 走 text node 搜尋反而穩定。
- **對策**：需要驗「列表有幾項」時，改驗多個已知文字存在（`get_by_text` 各項文字 visible）而非用 chained `count()`；或用 `page.locator("aside ul li a")` 不走 filter chain。
- **適用範圍**：凡是在 `filter()` 後繼續 chain CSS selector 並呼叫 `count()` 的情境。

## 2026-05-14｜src 的 CSS class 不能直接當 selector — Tailwind scoped class 在 DOM 中消失

- **現象**：`SectionList.vue` 中寫 `class="section-list"`，用 `page.locator(".section-list")` 找不到元素；`PreviewPanel.vue` 中寫 `class="preview-panel"`，同樣找不到。
- **原因**：Tailwind JIT build 會 purge 掉沒有對應 utility 的 class；Vue scoped CSS 產生的 class 也可能在 build 後變成 hash 或消失。**src 裡的 CSS class 名稱不會原封不動出現在 DOM 中**。
- **對策**：
  - ✅ 用語意型 selector：`get_by_text()`、`get_by_role()`、`get_by_label()`（這些從 src 讀文字 literal 可信）
  - ✅ 用結構型 selector：`<aside>`、`<ul>`、`<li>` 等 HTML 語意標籤
  - ❌ 不用 src 裡的 CSS class name（`.section-list`、`.preview-panel` 等）當 selector
  - ❌ 不用 img alt（src 寫 `alt="completed"` 但實際 DOM 是 `alt=""`）
- **實例**：2-3-11 `aside.filter(has_text="填寫建議")` 替代 `.section-list`；tick icon 改用 `aside ul li img`。
- **適用範圍**：所有從 Vue SFC src 解碼 class 名稱後直接用作 locator 的地方。

## 2026-05-14｜Windows 中文路徑：`!` bash 爛掉，要用 PowerShell 工具

- **現象**：在 Claude Code 用 `!` 前綴執行指令時，含中文字的 Windows 路徑（如 `D:\daily-records\專案\FOXCONN\...`）出現 `No such file or directory`，encoding 變成亂碼。
- **原因**：Claude Code 的 `!` 走 bash（POSIX shell），Windows 中文路徑在 bash 下 encoding 不相容。
- **對策**：一律改用 **PowerShell 工具**（不是 `!`），路徑用雙引號包住即可正常處理中文。
- **pytest 執行方式**：本專案 pytest 在 `.venv` 裡，`pytest` 指令不在 PATH，必須用完整路徑：
  ```powershell
  Set-Location "D:\daily-records\專案\FOXCONN\POC_for_autotest"
  $env:PYTHONUTF8=1
  .\.venv\Scripts\pytest.exe <test_paths> -v
  ```
- **適用範圍**：所有需要在 Claude Code 內執行 pytest / 任何 CLI 指令的場合，只要路徑含中文就用 PowerShell 工具。

## 2026-05-05｜selector 工作流程修正：src 解碼 ≠ 盲試

> 本條修正 2026-04-30「Token 消耗」對策 #1 與「ROI 評估」對策 #1（保留原文為歷史快照，反映當時的恐慌反應）。

- **4-30 教訓被誤讀**：真正的禁區是「AI 對 live DOM 試錯猜 selector」（盲試 + 高成本上下文累積）；被簡化為「AI 不能寫 selector，須由人 codegen」過度極端。讀 src 是**預先準備**，與盲試方向相反。
- **2026-05-05 實證**：2-2-2-B selector 工作以「讀 4 個 src 檔（`STFilter.vue` + `EVEventEntity.ts` + `PageFormItem.ts` + `STCommon.model.ts`）→ 寫 selector → pytest 驗證」流程完成，預估省 5–10 倍試錯。STFilter 渲染樣態的解碼成果可攤提到所有後續用篩選的工項。
- **src 讀取邊界**（避免重蹈 4-30「套用 vs 查詢」覆轍）：
  - ✅ src 寫**字面字串**（如 entity 的 `Name: "主辦單位"`）→ 可直接信任，DOM 渲染就是這字
  - ❌ src 寫**i18n key**（如 `Components.STFilter.Apply`）→ 仍需 DOM 確認實際渲染文字
  - 區別：字面字串無法被 runtime override；i18n key 可被 prop / runtime 覆蓋
- **新流程**（已寫入 `CLAUDE.md` 第 1 條，4 步）：
  1. 解碼 src（targeted 讀 view + entity + 共用 wrapper）
  2. 寫 selector（依 src 證據 + 第 3 條策略）
  3. pytest 驗證（紅綠是審查機制，不是再次盲試的觸發點）
  4. 紅了二修（使用者 dump 一頁 HTML，AI 對著真實 DOM 改；最多 3 輪）
- **使用者參與點**：原本「人工 codegen 取 selector」變為「pytest 紅時 dump HTML 一次」——使用者不需懂 selector 程式碼，只需執行 pytest + 必要時複製 page HTML。
- **影響範圍**：所有後續 selector 工作；2-2-2-C/D 沿用此流程。

## 2026-04-30｜Token 消耗巨大：AI 試錯法找 Selector

- **現象**：寫完第一批測試案例（2-2-2-B，4 個 case），光是 selector 除錯就幾乎用完一天的 token quota
- **原因**：採用「AI 猜測 selector → 執行失敗 → AI 探查 DOM → AI 修 → 再執行」的迴圈。每輪循環都累積大量 context：
  - Explore 子 Agent 回傳的報告過於冗長（程式碼＋表格＋分析，遠超實際所需）
  - headless DOM 探查腳本輸出大量無關噪音（checkbox 欄位×20）
  - 跨 session 的壓縮摘要本身就很大
  - 每次 pytest 執行輸出 50–100 行
- **對策**：
  1. **人工 inspect 優先**：新頁面先由人用 DevTools 或 `playwright codegen` 點一遍，確認所有 selector，再讓 AI 寫測試邏輯。AI 不應試圖替代這個步驟。
  2. **限制 subagent 回應長度**：呼叫 Explore agent 時明確要求「只回傳關鍵事實，200 字以內」。
  3. **一次報告所有問題**：執行失敗後，人工掃描輸出，彙整成一句話告訴 AI，而非讓 AI 逐一分析每個失敗。
  4. **分離探查與撰寫兩個階段**：先做 selector audit（人工），確認後再讓 AI 一次寫完所有 test file。
- **影響範圍**：P4 案例試點（2-2-2-B）

## 2026-04-30｜i18n 推斷與實際 DOM 不符

- **現象**：從 `zh-TW.json` 推斷「套用」按鈕文字是「查詢」，導致 `get_by_role("button", name="查詢")` selector 錯誤
- **原因**：i18n 檔的 `Components.STFilter.Apply = "查詢"`，但元件在實際使用時透過 prop 傳入自訂 label `"套用"`，覆蓋了 i18n。AI 從原始碼推斷，沒有以 DOM 為準。
- **對策**：**DOM 是唯一事實**。文字型 selector（`name="..."`, `has_text="..."`）一律以瀏覽器實際渲染的文字為準，不從原始碼或 i18n 推斷。若有疑問，用 `F12 → Elements → 複製文字` 確認。
- **影響範圍**：selectors.py 的 `apply_button`

## 2026-04-30｜PrimeVue 元件 ID 不穩定

- **現象**：DevTools「複製 selector」產生的 CSS path 包含 UUID（`#db0482f3-...`）與 `#pv_id_39`，這些 ID 在重新部署或頁面結構變更後會改變
- **原因**：
  - UUID 是 Vue 的動態 component instance ID
  - `pv_id_39` 是 PrimeVue 按元件掛載順序自動產生的 ID
- **對策**：
  1. 不使用 DevTools「複製 selector」產生的 ID-based CSS path
  2. 優先用 `get_by_role`、`get_by_label`、`get_by_text` 等語意型 selector
  3. 若必須用 class，選 PrimeVue 的語意 class（如 `.p-multiselect`、`input.p-datepicker-input`），避免 Tailwind utility class（如 `.gap-\[var\(--label-gap\)\]`）
  4. 長期解：請前端工程師在關鍵互動元件加上 `data-testid` 屬性，selector 即永久穩定
- **影響範圍**：selectors.py 所有元件 selector

## 2026-04-30｜UI 自動化測試 ROI 評估

- **現象**：導入自動化後仍需人工點過新頁面（用 codegen 或 inspect），被質疑與手動測試差異何在
- **結論（誠實評估）**：

  | 場景 | 結論 |
  |------|------|
  | 一次性驗收，之後不再迭代 | ❌ 手動測試效益更高 |
  | 每月有新版本上線 | ✅ 第 2 次 deploy 後開始回本 |
  | 只驗「元件存在」（smoke） | ⚠️ 價值有限 |
  | 驗完整業務流程（新增→查詢→驗證） | ✅ 高價值 |
  | 無 CI 整合（手動執行腳本） | ⚠️ 只節省部分時間 |
  | 有 CI（每次 deploy 自動跑） | ✅ 最大效益 |

- **關鍵認知**：自動化測試的效益來自**回歸保護**，不是驗新功能。新功能的 selector discovery（30 分鐘）是一次性入場費；舊功能每次 deploy 不再手動驗，才是持續回報。
- **對策（正確分工）**：
  1. 人工：codegen 點一遍新頁面，取得穩定 selector（每個新功能一次）
  2. AI：用確認的 selector 寫所有測試邏輯（不再做 selector 探查）
  3. CI：每次 deploy 自動執行（沒有 CI 則效益大幅縮減）
- **長期建議**：
  - 前端加 `data-testid` → selector discovery 成本趨近於零
  - 只寫 3–5 個核心 E2E happy path，不追求全 AC 覆蓋
  - 測試覆蓋業務流程（create → query → verify），而非元件存在性

- **影響範圍**：整個 POC 策略方向

## 2026-04-30｜Playwright 1.48 不抓 sessionStorage

- **現象**：`storage_state()` 儲存後，重新載入 session 時頁面仍跳回登入頁
- **原因**：Playwright 1.48 的 `storage_state()` 只抓 cookies 與 localStorage，不包含 sessionStorage。此系統的 auth token（`backendtoken`、`xsessionCode`）存在 sessionStorage。
- **對策**：
  1. warm-login 結束時額外執行 `page.evaluate("JSON.stringify(sessionStorage)")` 並存成 `.auth/admin.session.json`
  2. conftest.py 的 autouse fixture 用 `context.add_init_script(js)` 在每個頁面 navigate 前還原 sessionStorage
- **影響範圍**：`lib/auth.py`、`tools/warm_login.py`、`conftest.py`

## 2026-04-30｜PowerShell 中文輸出亂碼

- **現象**：pytest 在 PowerShell 執行時，中文測試名稱與訊息顯示為亂碼（cp950 vs UTF-8）
- **原因**：Windows PowerShell 預設 cp950 encoding，Python/pytest 輸出 UTF-8
- **對策**：執行前設定 `$env:PYTHONUTF8=1` 或 `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8`，或改用 Git Bash 執行（天然 UTF-8）
- **注意**：此問題不影響測試邏輯與報告檔案（報告以 UTF-8 寫檔），僅影響終端機顯示
- **影響範圍**：開發體驗，非功能性問題
