##初始提示詞

我現在要針對一個網站開發專案導入自動化測試

範圍 
```
	input/WBS.md #測試的工項編號與標題
	需求規格.pdf #需求規格描述
```

工項編號與規格描述文件的對應方式

規格
```
2-2-2 搜尋功能  
2-2-2-A 根據登入者綁定活動過濾  
2-2-2-B 主辦單位 / 活動日期 / 活動名稱篩選  
2-2-2-C 重置按鈕清空所有篩選  
2-2-2-D 查詢按鈕觸發查詢
```

描述中的段落分法會是
```
2-2 搜尋功能
1.XXXX
2.XXXX
A.XXXX #對應2-2-2-A 根據登入者綁定活動過濾 
B.XXXX #對應2-2-2-B 主辦單位 / 活動日期 / 活動名稱篩選  
C.XXXX #對應2-2-2-C 重置按鈕清空所有篩選  
D.XXXX #對應2-2-2-D 查詢按鈕觸發查詢
```

測試網站
[受測站登入頁](https://<受測站 host>/entry/login)
測試帳密要另外用設定檔案設定，有圖形驗證碼要想辦法繞過或啟用測試的時候可以人工輸入

- 我需要自動化的工具可以自行操作瀏覽器進行測試，不會給你原始碼
- 我每次測試要依據單次啟用測試所提供的編號進行指定腳本的測試
- 要幫我生成測試案例腳本
	- 測試完要產生md檔案的測試報告，檔案名稱要有測試報告要有YYYYMMDD HH:MM
	- 一個工項包一組測試案例檔案，測試案例中可以存再多個子案例，用一個目錄歸納所有測試案例檔案
	- 測試案例有可能被我審查修訂
	- 測試案例的項目有可能被我增刪修
	- 測試案例項目如果有不符合預期的要列出包含但不限於以下欄位(你可以依據規劃建議擴充項目)
```
測試案例編號：XXXX
工項編號：2-2-2-A
實際效果：OOOO
預期效果：XXXX
URL：XXXX
```
- 要幫我生成需求規格
	- 需求規格要幫我從PDF展開成md
	- 一個檔案不要太大，一個工項包一組需求規格檔案，需求規格中可以存在多個子規格，用一個目錄歸納所有需求規格檔案
	- 檔案名稱要完全對應工項描述，方便複製檔案名稱後回到原本的WBS操作關鍵字搜尋
	- 需求規格工項可能增刪修
	- 需求規格文字也可能增刪修

我想像的如上，我需要你
- 跟我確認我沒說明清楚的需求
- 評估執行可行性或障礙點
- 幫我制定計畫我審查後再執行實作
- 要包含自動化測試的專案目錄檔案架構設計給我參考
- 各檔案要有範例，搜錄我有明確定義的提示詞及你建議生成的
- 幫我選擇測試工具
- 實作步驟與環境
	- 我的筆電是win11，但我也有WSL2的linux環境
- 如果可能我希望這個過程的提示詞及重點經驗可以被記錄，以便我未來可以將這樣的工程化方法重現再不同專案或典範轉移給其他成員
